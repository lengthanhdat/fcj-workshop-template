# Identity and Access Management (IAM)

Setting up Identity and Access Management (IAM) is the first and most crucial foundation to ensure that services within the **AuraAcademic** architecture can securely communicate with each other, and the CI/CD system (GitHub Actions) has sufficient permissions to update the source code.

In this step, we will create an **IAM User** for GitHub Actions and ensure the **ECS Task Execution Role** exists.

---

### Step 1: Create IAM User for GitHub Actions

GitHub requires a credentialed account to automate the process of pushing Docker Images to ECR and commanding ECS to update the application.

1. Log in to the **AWS Management Console**, search for `IAM` and select **IAM**.
2. On the left menu, select **Users**, then click **Create user**.
3. In the **User details** section:
   - User name: enter `github-actions-deploy`.
   - Click **Next**.
4. In the **Set permissions** section:
   - Choose **Attach policies directly**.
   - Search for and check the following policies:
     1. `AmazonEC2ContainerRegistryFullAccess` (Permission to interact with Docker Image repository - ECR)
     2. `AmazonECS_FullAccess` (Permission to update applications on ECS)
     3. `AmazonS3FullAccess` (Permission to upload Frontend files to S3)
     4. `CloudFrontFullAccess` (Permission to invalidate CDN cache after uploading new frontend)
5. Click **Next** to the Review page, then click **Create user**.

![Create IAM User for GitHub Actions](../../images/6-Workshop/6.1-IAM-Security/1-create-user.png)

---

### Step 2: Create Access Key (Credentials)

After successfully creating the User, we need to generate access keys for GitHub.

1. Click on the user `github-actions-deploy` you just created.
2. Switch to the **Security credentials** tab.
3. Scroll down to the **Access keys** section, click **Create access key**.
4. Select **Third-party service**, check the confirmation box, then click **Next**.
5. Click **Create access key**.
6. **IMPORTANT:** The screen will display the `Access key ID` and `Secret access key`.
   - Open Notepad/Text editor and copy these 2 keys to save them.
   - _The Secret access key cannot be viewed again after you click Done!_

![Access Key created successfully](../../images/6-Workshop/6.1-IAM-Security/2-access-key.png)

---

### Step 3: Verify ECS Task Execution Role

When the Backend application runs on Amazon ECS Fargate, it needs to borrow an AWS Role to have permissions to: Pull Docker Images from ECR and write Logs to CloudWatch.

1. On the left menu of the IAM screen, select **Roles**.
2. In the search box, type `ecsTaskExecutionRole`.
   - **Case 1:** If this role already exists (auto-generated by AWS previously), click on it and verify it has the `AmazonECSTaskExecutionRolePolicy` policy. If it does, you are done!
   - **Case 2:** If it does not exist, click **Create role**. Select Trusted entity type as **AWS service**, Use case as **Elastic Container Service** -> **Elastic Container Service Task**. Search for and attach the `AmazonECSTaskExecutionRolePolicy` policy. Name the role `ecsTaskExecutionRole` and create it.

![Configure ecsTaskExecutionRole](../../images/6-Workshop/6.1-IAM-Security/3-ecs-role.png)
